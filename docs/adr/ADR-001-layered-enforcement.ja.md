# ADR-001: 予測ベース分類から階層型エンフォースメントへの移行

**Status:** Proposed
**Date:** 2026-06-11
**Deciders:** kaz(メンテナ)
**関連:** [`ROADMAP.md`](../ROADMAP.md), [`SECURITY.md`](../../SECURITY.md)

## Context(背景と力学)

agent-belay の設計思想は「静的 denylist は壊れている — リストは文脈を読めない」
(README)。しかしこの批判は対称で、**whitelist にも同じ強度で適用される**。
実際、v0.3.x のコード精査では whitelist 側の同型バグが見つかっている:
`find` を read-only リストに載せた結果 `find -exec rm` が allow になる、
`curl -H "Authorization: …"` が素の `curl` より緩くなる、など。

根本原因はリストの品質ではなくアーキテクチャにある。現在のゲートは
「コマンドテキストから効果を**事前予測**する」構造であり:

1. 任意コードの効果の完全な事前判定は原理的に不可能(Rice の定理)。
   予測ベースである限り、知識構造=リストが必須で、穴が必ず残る
2. LLM 支援(v0.5 計画)も「暗黙の学習済みリスト」への置換であり、
   この限界を超えない
3. 現構造ではリストの穴=安全性の穴になる(fail-closed 化後も、
   予測層が境界を担う事実は変わらない)

一方、ロードマップの defining question は「guardrail か boundary か」であり、
v1.0 は「ユーザーが体重を預けられる保証」を目標にしている。予測の延長線上に
boundary はない。

## Decision(決定)

エンフォースメントを 4 層に再構成し、**安全境界を「予測」から
「封じ込め(チョークポイント)と観測」へ移す**。

```
L1 封じ込め   — エグレス制御・サンドボックス・OS権限。deny-all が境界。リスト不要
L2 観測判定   — スナップショット上で実行し、実測された効果(diff)を判定する
L3 予測判定   — 現在の分類器。境界ではなく、L4を煩くしないノイズ削減キャッシュ
L4 人間の承認 — 残った不確実性の最終受け皿(既存の one-shot 承認ループ)
```

- **外部効果**(不可逆・巻き戻し不能)は L1 で扱う: コマンド名の予測ではなく、
  ネットワークのチョークポイントで「観測された接続要求」を allow/deny/承認する
- **ローカルな破壊的効果**は L2 で扱う: スナップショット(worktree / CoW)上で
  実行し、実際の変更 diff を判定してからコミットする
- L3 のコマンドリストは維持するが、verdict の最終根拠ではなく
  「明白に安全なものの素通し」と「チョークポイント到達前の早期警告」に格下げする
- belay の中核資産は分類器ではなく**承認ループ+監査+運用 CLI**であり、
  L1/L2 に対する「承認ブローカー」として再配置する

### ホワイトリストは不要になるか(明示的に答える)

**不要にはならない。ただし役割が変わる。** 残るリストは 3 種:

1. **効果カテゴリのポリシー**(repo 外書き込み / ネットワーク送信 / 削除 /
   control-plane 変更)— 小さく、閉じていて、安定
2. **承認キャッシュ** — 人間が承認した接続先・操作の蓄積。設計時に書くリスト
   ではなく、運用の中で人間の判断から育つリスト
3. **L3 の予測リスト** — レイテンシ/ノイズ最適化としてのみ残存

決定的な変化はリスト外の帰結である。チョークポイントが deny-all である限り、
**ホワイトリストの漏れ=摩擦(人間に聞く)であって、侵害ではない**。
リスト品質は UX を左右するが、安全性を左右しなくなる。これにより README の
denylist 批判は自家撞着でなくなる。

## Options Considered

### Option A: 予測の高度化(現路線の延長)

リスト拡充 + 引数解析 + LLM 支援で予測精度を上げ続ける。

| Dimension | Assessment |
|-----------|------------|
| 複雑性 | 中(漸進的だが終わりがない) |
| コスト | 低(既存構造のまま) |
| 到達可能な保証 | guardrail 止まり。リストの穴=安全性の穴が恒久化 |
| プラットフォーム依存 | なし |

**Pros:** 既存コードの延長。インストール体験が変わらない。
**Cons:** 理論的上限が低い。バイパス発見→修正の競争が恒久化。
製品の中心的主張(denylist 批判)と自家撞着のまま。

### Option B: 純粋な OS サンドボックス(belay の解体)

エージェントをコンテナ/別ユーザーで隔離し、ゲート製品をやめる。

| Dimension | Assessment |
|-----------|------------|
| 複雑性 | 高(インフラ製品化) |
| コスト | 高(既存資産の大半を放棄) |
| 到達可能な保証 | boundary(adversarial 含む) |
| プラットフォーム依存 | 大 |

**Pros:** 最強の保証。判定問題そのものが消える。
**Cons:** deny-all は承認 UX がなければ使われない。粒度の細かい
「この 1 回だけ許可」の運用(belay の強み)が失われる。既存の
hook アダプタ・承認ループ・監査資産を捨てることになる。

### Option C: 階層型(L1–L4)— belay を承認ブローカーへ(採用)

| Dimension | Assessment |
|-----------|------------|
| 複雑性 | 中〜高(段階導入可能) |
| コスト | 中(L3/L4 資産は全て再利用) |
| 到達可能な保証 | 層ごとに明示できる boundary(L1/L2)+ guardrail(L3) |
| プラットフォーム依存 | L2 のバックエンドのみ(抽象化で吸収) |

**Pros:** 承認ループ・監査・アダプタ資産が L1/L2 の UX 層としてそのまま効く。
opt-in で段階導入でき、各層が独立に価値を持つ。保証を層別に正直に語れる。
**Cons:** インストール体験が変わる(プロキシ/サンドボックス連携)。
プラットフォーム差分の保守。レイテンシ予算の管理が必要。

## Trade-off Analysis

- **A vs C**: A は限界費用が逓増する(リスト保守は終わらない)のに対し、
  C は L1 のエグレス制御だけで「外部効果の列挙問題」が消える。
  外部効果は belay が deny する対象の中で最も重要かつ最も列挙困難であり、
  ここをチョークポイント化する効果対費用が最大
- **B vs C**: B の deny-all は承認体験がなければ現場で無効化される
  (「ノイズの多いゲートは切られる」原則)。C は B の強制力に belay の
  承認 UX を接続するもので、対立ではなく分業
- **残るリスク**: covert channel(DNS 経由漏出等)は L1 でも残る。
  same-OS-user の adversarial エージェントは v0.9 の control-plane 分離
  まで対象外。いずれも SECURITY.md に層別の保証として明記する

## Consequences

- 容易になること: 保証の説明(層ごとの guarantee table)。外部効果の
  網羅(列挙→観測)。reversibility / blast radius の実測化(L2 の diff)。
  README の思想と実装の一致
- 困難になること: クロスプラットフォーム保守(L2 バックエンド)、
  インストールの段階化(L1 はインフラ前提を持つ)、レイテンシ予算管理
- 再訪が必要になること: v0.5 の「予測高度化」項目の優先度
  (L3 が境界でなくなるため、コーパス測定は維持しつつ投資は縮小可)。
  `SECURITY.md` の threat model 記述。fail-closed デフォルト(v0.4)の
  意味付け(L1 導入後は「L1 未設定環境のフォールバック」になる)
