# ADR-002 — Concept conformance: すべてのルールは restorability floor に奉仕する

Status: Accepted（拘束力のある運用規律）
Date: 2026-06-13
Context: egress 過剰ブロック（v2.1.3）を起点に、コードがコンセプトから繰り返し
ずれていた問題への再発防止。

---

## 1. 何が起きたか（incident pattern）

belay のコンセプトは **restorability floor**:
> 「取り消せるなら通す。**取り消せない × 破滅的**だけ止める。98% は黙過。fence ではない。」

にもかかわらず、コードには**コンセプトに反するルールが堆積**していた:

- `curl https://example.com`（無害な GET）を ask → 何も変えず・送らず・取り消すものが無いのに止める。
- `aws s3 ls` / `gh pr list` / `kubectl get` まで ask → ツールを丸ごと external 扱い（fence）。
- これらは「ツールを止める」(fence) であって「不可逆な行為を止める」(floor) ではない。

そして対処も **症状パッチ**（v2.1.2＝偽陰性、v2.1.3＝偽陽性）に流れ、「**なぜ毎回ずれるのか**」を
問えていなかった。

## 2. 根本原因（root cause）

1. **コンセプトが「契約」として強制されていない**。[CONCEPT.md](../CONCEPT.md) は散文として存在するが、
   コードのルールがそれに**準拠していることを検証する仕組みが無い**。
2. **テストが FN だけを門番している**。構造スイートは「破滅を見逃さない(MUST-ASK)」を硬く守るが、
   「**復旧可能を止めない(MUST-ALLOW)**」を同格には守っていなかった → **fence 化（偽陽性）が
   テストに映らない**。
3. **ルールに正当化が無い**。`TIER0_EXTERNAL_KEYS` に `curl`/`aws` を足す時、「これは不可逆×破滅的か?」
   を誰も問わなかった。
4. **再定義時に旧ルールを監査しなかった**。v0.7「egress チョークポイント(fence)」を v2.0
   「restorability floor」に作り直した際、矛盾する旧ルールを刈り取らなかった。
5. **貢献者(AI 含む)が症状から推論した**。コンセプトを起点にせず、目の前の不具合を局所修正した
   （本セッションで AI 自身が egress 一律 ask を一度容認したのもこれ）。

## 3. 不変の原則（the single rule）

belay が作っているものは **この1つの判定規則**であり、すべてのコード/仕様はこれに奉仕する:

> **その操作が「間違い」だったとして、取り消せるか?**
> - 取り消せる（または何も変えない）→ **通す**。
> - 取り消せない **かつ** 破滅的 → **ask**。
> - 判定が曖昧 → Tier1（fail-closed = ask）。

これに当てはまらない理由で止めるルールは、**定義上 fence であり、belay ではない**。

---

## 4. 再発防止策（measures — 拘束力あり）

### M1 — コンセプトを単一の判定規則として明文化し、全 Tier0 ルールをそれで正当化する
- [CONCEPT.md](../CONCEPT.md) に §3 の判定規則を**唯一の真実**として置く。
- すべての Tier0 決定論ルールは、この規則からの帰結でなければならない。

### M2 — MUST-ALLOW を MUST-ASK と**同格の硬いゲート**にする（最重要）
- 構造スイートの `MUST-ALLOW` を、FN ゲートと同等の**ハード CI ゲート**に格上げする。
- benign/復旧可能コーパス（読み取り・一覧・無ペイロード GET・取り消せるローカル操作）が
  **1 件でも ask されたら CI 落ち**。これで fence 化（偽陽性）が FN と同様に可視・不可避になる。
- **対称性**: FN=0（破滅を見逃さない）と FP→0（復旧可能を止めない）は**両輪**。belay の価値は
  「**止める範囲の狭さ**」にあり、FP は単なる UX 問題ではなく**コンセプト違反**である。

### M3 — Tier0 ルールごとに justification を必須化
- Tier0 のリスト/分岐に新規追加するルールは、**「なぜ不可逆×破滅的か」を1行で正当化**する
  （コメント or 仕様）。
- 正当化できないルールは Tier0 に置かない（→ 削除、または Tier1=fail-closed へ委譲）。
- 各 Tier0 ルールには **MUST-ALLOW の反例**（それが過剰に止めないことを示す近傍の benign 例）を
  ペアで用意する（M2 のコーパスに入れる）。

### M4 — 仕様/PR に concept レビューゲート
- 新しい要件・ルールは、レビュー時に次を明示的に答える:
  - [ ] これは**復旧可能な操作を止めていないか**?（止めていれば設計が間違い）
  - [ ] 止める根拠は「**不可逆 × 破滅的**」か?（ツール名・カテゴリで止めていないか）
  - [ ] 曖昧ケースは Tier1(fail-closed) に落ちるか?
  - [ ] MUST-ALLOW / MUST-ASK の両方にテストを足したか?

### M5 — コンセプト再定義時の全ルール監査を必須化
- コンセプト/設計を作り直す（v0.7→v2.0 のような）時は、**既存ルールを新コンセプトで総点検し、
  矛盾を刈り取る**ステップを必須の作業項目にする。今回の egress 残存はこの監査欠落が原因。

### M6 — 「fence ではない」を測る
- silent-pass 率 / 偽陽性率を dogfood で可視化・追跡する。「98% 黙過」が実測で崩れていないかを
  定期確認（egress の過剰 ask は本来ここで早期に検知されるべきだった）。

### M7 — 貢献者(AI 含む)の規律
- 不具合は**症状からでなくコンセプトから推論する**。「この修正はコンセプトのどの帰結か」を
  先に言語化してから書く。コンセプトに照らして説明できない変更は出さない。

---

## 5. 適用（first application）

- **v2.1.3 egress 過剰ブロック修正**は M2/M3 の最初の適用。read→allow / mutate・exfil→ask /
  不明→Tier1 の三分岐 + MUST-ALLOW/MUST-ASK 両系列テストは、本 ADR の規律そのもの。
- 実装時に **M2（MUST-ALLOW ハードゲート）を構造スイートへ恒久化**し、以後すべての Tier0 変更が
  この両輪ゲートを通る状態にする。

## 6. 一行で

**belay が作っているのは「取り消せない×破滅的だけを止める床」。それ以外を止めたら、それは belay
ではなく denylist だ。** すべてのルール・仕様・修正は、この一文で説明できなければ出さない。
