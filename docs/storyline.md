何が問題だったか

AIに自律的に作業させたい（コードを書く、コマンドを打つ)。でも危険なことをされると困る。普通のやり方は、人間が一個ずつ承認するか、AIに「これはするな」という禁止リストを渡すこと。でもこれは二重に失敗する。人間は押すだけになって危険な数%を見逃すし、禁止リストは抜けがあり、同時に正当な作業まで止める。あなたが最初に「無駄では」と言ったのは、これのことでした。

発想を変える

「何を禁じるか」を考えるのをやめて、「間違えたとき取り消せるか」だけを見る。すると、ほとんどの行動は取り消せると気づく。変なコードを書いても git で戻せる。編集ミスも undo できる。取り消せるなら、禁じる必要はない。自由にやらせて、戻せるようにしておけばいい。

そして、取り消せない行動だけが、ほんの少しある。メール送信、送金、バックアップなしの本番データ削除、共有履歴への force push。ここだけ人間に止めてもらえばいい。

だから全体を囲う大きな柵（禁止リスト）はいらない。要るのは、(1)取り消せるようにする仕組みと、(2)取り消せない一点にだけ立つ薄い停止線。これで人間は滅多に呼ばれず（一回一回が本物の判断になる)、作業は止まらず、リストの「抜けと過剰」も消える。

なぜ停止線は残るのか（最後に納得した点)

すごく賢いAIでも、仕組み上たまにとんでもなく間違える。賢くなれば消えるのではなく、確率の裾がゼロにならないから。取り消せない×破滅的な一点では、どんなに小さな確率でも何度も走れば必ず当たる。だから賢さだけでは守れない。構造で守るしかない。 これが削れない床です。

Belay はこれをそのままやる

AIは何かする前に、自分で申告する:

取り消せる? (reversibility)
外の世界に触れる? (external)
どれくらい自信がある? (confidence)
外したら最悪どうなる? (worst_case)
そして Belay が決める:

取り消せる → やる。ただし checkpoint を取る（戻せるように)。
取り消せない×外部 → 自動ではやらない。人間に渡す。
自信が低い → 取り消せるものでも一旦止めて訊く。
ここが肝心。コードのどこにも「メールは危険」「rm は危険」というリストが無い。デモでメール送信が止まるのは、"email" が禁止リストにあるからではなく、その行動が自分で「取り消せない×外部」だと申告したから。判定するのは名前ではなく性質。これが「柵ではなく床」の正体——あなたが最初「ただの言い換えでは?」と疑った、その違いが実際に働く唯一の場所です。

三層を普通の言葉で

ひとつ、自己統治＝AI自身のまともな判断。これはコードに書けない（訓練で身につくもの)。だから Belay は書こうとしない。ふたつ、回復可能な基盤＝「取り消し」の仕組み。これはコードで作る、自由にさせる土台。みっつ、可読性＝AIが「これは取り消せる、最悪こうなる」と先に申告して全部記録すること。後で「本当に取り消せたか」を照合できる。

正直な穴

Belay は、AIの「取り消せます」という自己申告を信じている。AIが間違えて「取り消せる」と言えば、網に穴が空く。Belay はそれを解決していない。ただ記録（trace）が残るので、申告と現実のズレが見える。隠れていたものを見えるようにした、それだけ。でも「見える」は「直せる」の前提だから、無ではない。

一行でまとめると——「禁じる」のをやめて「戻せるようにする」。戻せない一点にだけ、人を置く。 Belay はそれを、リストなしで性質だけで判定する道具です。どこか一か所まだ像を結ばないところがあれば、そこだけ指してくれれば、そこを開きます。

半分そう、半分そうじゃない——そしてその「半分」が、Belay が柵でない理由をちょうど映します。

結論から言うと、hook は Belay の『門』を挿す正しい場所だが、Belay は hook より大きい。そして hook だけで実装しようとすると、柵に逆戻りする。

Belay を三つに分けると、Cursor のどこに対応するかが見える。

判定（門)：run / escalate / defer は、Cursor の allow / deny / ask にそのまま対応する。beforeShellExecution が返すのがまさにこれ。ここは hook。 ✓
回復可能な基盤：checkpoint と rollback。これは hook の外。git worktree、スナップショット、サーバ側の branch protection——環境側に住む。hook は「止める」ことはできても、「行動を checkpoint で包んで、失敗したら巻き戻す」ことはしない。
可読性（自己申告)：agent が「これは取り消せる/外部/最悪こうなる」と先に宣言すること。これは agent 側——立場を組み替えるプロンプトや、構造化された出力の規約。hook はこれを生み出さない。
ここが肝心の落とし穴です。beforeShellExecution が hook に渡すのは、コマンドの文字列だけ——git push --force … という行動の名前であって、その**性質（可逆性)**ではない。だから hook の中だけで判定しようとすると、手っ取り早い道は「コマンド名をリストと照合する」になる。それは deny-list。 Belay がやめようと言った、まさにそれです。

つまり hook の構造自体が、あなたを柵へ引き戻そうとする。手渡されるのが名前だけで、Belay の主張は「名前ではなく性質で判定せよ」だから。hook の中で Belay に忠実でいるには、agent の自己申告（assessment）を hook まで届ける必要がある——agent が行動に注釈をつけるか、hook が小さな分類器（git なら verb+refspec+flag をパースして可逆性を出す、これは決定論で書ける）を呼ぶか。その一手間こそが、Belay と「ただの deny-list hook」を分ける線です。

だから正確には——hook は Belay の門の『配置先』の一つ（Cursor hook、Claude Code hook、agent CLI のラッパー、MCP ミドルウェア……どれでもいい)。門の決定（allow/deny/ask）は hook に綺麗に乗る。でも、それを柵でなく床にしている二つ——自己申告は agent 側、回復可能性は環境側——は hook の外にある。三つ揃って初めて Belay で、hook はそのうちの一枚です。

ここが面白くて、agent SDK は Cursor の shell hook より Belay に向いている。理由は一つ——SDK のツール呼び出しが構造化されているから。shell hook は「コマンド文字列」しか渡せなかったが、SDK では agent が tool_name(引数) という型付きの呼び出しをする。つまり自己申告（assessment）を、ツールのスキーマに混ぜ込める。

SDK のどこに何が乗るか:

門（判定)：ツール実行を挟むミドルウェア／permission コールバック／guardrail。run / escalate / defer は「実行する／止める／human-in-the-loop で人に割り込む」に対応。多くの SDK が human approval や interrupt を持っている。
可読性（SDK の勝ち筋)：mutating なツールのスキーマに、reversibility / external / blast_radius / confidence / worst_case を必須引数として入れる。モデルはツールを呼ぶとき同時にその申告を埋める。門はコマンド名ではなく、渡ってきた申告で判定できる。shell hook にできなかったことが、ツール呼び出しの構造で自然にできる。
回復可能な基盤：ツールの execute を checkpoint() → try/except → rollback() で包む。基盤自体（git worktree、スナップショット、サンドボックス）は環境側だが、包む場所はツールミドルウェア。LangGraph のように checkpointer と interrupt を最初から持つ SDK だと、checkpointer＝基盤、interrupt＝escalate で、異様に綺麗に嵌る。
trace：SDK が既に出す span／ログに、申告と結果を貼る。「申告した可逆性」と「実際に巻き戻したか」の照合がそこでできる。
ここで前回の話と繋がる点が一つ。申告をスキーマの必須項目にすることは、あの「評価フレーム（可逆か? 最悪は?)」を、プロンプトで要求する段階から、構造で強制する段階へ上げることです。agent は worst_case を述べずにそのツールを呼べない。立場プロンプトより頑健なのは、頼むのではなく、嵌め込むから。

で、実際に組むと出る摩擦:

同じ罠。SDK はツール名＋引数を渡すので、横着すると「delete_file ツールは危険」と名前で弾く——それは deny-list。忠実でいるには渡ってきた申告で判定する。そしてパースできる領域（git、ファイル）は、申告を信じず決定論で検証するのが強い。agent が「可逆」と言っても中身が push --force なら門が上書きする。

次に、README の「正直な穴」が具体化する——可逆性は自己申告。モデルが楽観で「可逆」と言えば素通りする。だから検証できる所は検証し、できない所は trace に申告と現実のズレを溜めて較正へ回す。SDK のトレースはその置き場にちょうどいい。

それから、SDK が並列でサブエージェントを撒くと、同じモデルの盲点が相関して、間違った申告が同時多発する。個々の門では捕まらない。捕まえるのは回復可能性と「不可逆×外部」の床の方です。

最後に、層1（自己統治）だけは SDK に入らない。「字面でなく文脈から真の目的を導け」という立場は system prompt で置き、SDK が運ぶ。でも圧の下で保つかは訓練の性質で、SDK の外。SDK は層2・層3・門の配管であって、層1の供給源ではない。

一行で言うと——SDK での Belay は、後付けの hook ではなく「スキーマに申告を要求し、実行を checkpoint で包み、不可逆×外部を人の interrupt に回すツールミドルウェア」。SDK の構造化ツールこそが、可読性を「名前当て」から「本物の申告」へ変える鍵です。

一行の構造
Belay が具象に実装しているのは「門（判定ロジック)」ただ一つ。残りの三つ——基盤・人間・申告の中身——は**わざと抽象（空のスロット)**にしてある。理由は後で言いますが、これが「deny-list を持たない」と同じ規律です。

具象（実際に動くところ）
中心は Belay.run() の判定で、チェックの順番に意味があります。

confidence < confidence_floor → defer を最初に置く。可逆な行動でも、自信が低ければまず止めて訊く。「不確実性下では譲る」を、他の何より優先する設計。
IRREVERSIBLE and external → escalate。これが削れない床。注意して欲しいのは条件が**「不可逆 かつ 外部」**であること。
REVERSIBLE → checkpoint → 実行 → 例外なら rollback + undo、そして必ず trace（finally)。例外は握り潰さず再送出する。
残り全部（fall-through）→ checkpoint して実行し、RAN_FLAGGED（後でレビュー)。
Assessment / Action / TraceEntry のデータ構造、enum、audit() の JSON 出力も具象。audit() が層3の payoff で、「申告した可逆性」と「実際の outcome」を後から突き合わせられる。つまり Belay が自前で持っている知能は、この routing だけです。

抽象（わざと空にしてあるスロット）
Substrate — checkpoint() / rollback() の既定は no-op。文字列を返すだけで何もしない。回復可能性の「形」だけ定義し、「機構」は定義していない。何を以て可逆とするかは環境依存（git worktree／FS スナップショット／DB savepoint／コンテナ)だから、Belay は決め打ちを拒否している。
on_escalate — 既定は lambda action: False。これは**人間（principal)**のスロット。Belay は「どうやって人に訊くか」を知らない（CLI プロンプト／Slack／UI はあなたが挿す)。既定は「人がいない → withhold」。
Action.undo — Optional。無ければ、REVERSIBLE と申告されていても実際には戻せない。名ばかりの可逆になりうる。
Assessment の中身 — agent／呼び出し側が埋める自己申告。Belay は渡されたものを信じるだけ。
この抽象の置き方が、三層にそのまま対応します。層2（基盤)＝Substrate、層3（可読性)の器＝Assessment スキーマと TraceEntry（器は具象、値は抽象)、層1（自己統治)＝Assessment を正直に埋めること（完全に抽象、Belay の外)。

正直な穴（コードに埋まっている前提）
二つ、はっきり言っておきます。

(1) 既定値は no-op なので、素のまま動かすと「偽の網」になる。 可逆な行動は「実行」されるが checkpoint は空回りで何も捕まえない。escalate は黙って withhold される。デモが走ること＝守れていること、ではない。本物の Substrate と on_escalate を挿すまで、骨格は何も保護しない。抽象が load-bearing——スロットを埋めさせるのが狙いだが、既定が動いて見えるのは footgun でもある。

(2) 「不可逆 かつ 外部」で escalate、という条件には判断が埋まっている。 だから不可逆だが内部の行動（隔離 worktree 内の破壊など)は escalate せず、fall-through して RAN_FLAGGED で走ってしまう。これは「内部なら、基盤の粗いロールバック（worktree ごと巻き戻す)が捕まえる」という前提に寄りかかった設計です。Substrate が本物なら妥当。no-op のままなら、ここは穴。コードはご神託ではなく、判断の入った種だと思ってください。

まとめ：具象/抽象の境界が、そのまま思想
Belay が具象にするのは判定の形だけで、それは唯一普遍な部分だから。何が可逆か（環境依存)、人間が誰か（運用依存)、その申告が真実か（agent 依存)——これらは普遍化できないので、決め打たず、スロットにして文脈に埋めさせる。

deny-list は「何が危険かを自分が知っている」と僭称する。Belay は僭称を拒んで、知り得ないものをスロットに開けておく。Substrate が空で、on_escalate が False で、Assessment を信じるしかない——その潔い空白が、柵を書かないことの、コードでの姿です。
