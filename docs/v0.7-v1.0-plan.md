# v0.7〜v1.0 計画 — 階層型エンフォースメントへの移行

根拠: [`ADR-001`](./ADR-001-layered-enforcement.md)(予測ベース分類から
L1–L4 階層型エンフォースメントへ)。
前提: [`v0.5-plan.md`](./v0.5-plan.md)(判定パイプライン・コーパス測定)と
[`v0.6-plan.md`](./v0.6-plan.md)(監査ツーリング・帯域外承認・署名トークン)が
出荷済みであること。特に v0.6 の署名付き承認トークンと通知チャネルは
v0.7 以降のブローカー UX の土台になる。

```
L1 封じ込め   … v0.7(エグレス)+ v0.9(capability / control-plane 分離)
L2 観測判定   … v0.8(トランザクショナル実行)
L3 予測判定   … 既存分類器。v0.7 以降「ノイズ削減キャッシュ」に格下げ
L4 人間の承認 … 既存承認ループ。全層のエスカレート先として不変
```

各マイルストーンは opt-in で出荷し、単独で価値を持つ。デフォルト化は
それぞれ dogfood 計測(v0.3.1 `metrics` の延長)を条件とする。

---

## v0.7 — Egress chokepoint(外部効果の列挙をやめる)

**テーマ:** 最も列挙困難で最も重要な「外部効果」を、コマンド名の予測から
ネットワークチョークポイントでの観測に移す。ADR-001 で効果対費用最大とした層。

- **ローカルエグレスプロキシ**: エージェントプロセスの HTTP(S) 通信を
  belay 管理のプロキシ経由に強制する(環境変数注入 + 直接続の検出)。
  プロキシは「どのプロセスがどの接続先へ」という**観測事実**に対して
  allow / deny / 承認エスカレートを判定する
- **承認ブローカー化**: deny された接続は既存の one-shot 承認ループに乗せる
  (`/belay-approve`、TTL、revoke、監査記録は R21 セマンティクスのまま)。
  v0.6 の帯域外承認・署名トークンをそのまま使う
- **接続先ポリシー = 承認キャッシュ**: ドメイン allowlist は設計時に書く
  リストではなく、人間の承認から育てる(承認時に「今回のみ / このドメインを
  恒久許可」を選択)。ADR-001 の「残るリスト②」の実装
- **L3 の格下げ第一弾**: `EXTERNAL_COMMANDS` リストは「プロキシ未設定環境の
  フォールバック」と「実行前の早期警告」に役割変更。プロキシ有効時は
  外部効果の境界を L1 が担うことを `explain` / SECURITY.md に明記
- **限界の明文化**: DNS 漏出等の covert channel、プロキシを迂回する raw
  socket は対象外(adversarial は引き続き L1 完全版 = v0.9 以降 + OS 制御)

**終了条件:** プロキシ有効環境で、未知のコマンド経由
(`python -c`、独自バイナリ等)の外部送信が接続時点で捕捉・承認フローに乗る。
コマンド名リストに依存した外部効果判定がゼロになる(コーパスで実証)。

## v0.8 — Observed effects(実行してから判定する)

**テーマ:** ローカルな破壊的効果を「予測ラベル」から「実測 diff」へ。
reversibility / blast radius が初めて測定値になる。

- **トランザクショナル実行**: 低確信(L3 の較正済み confidence が閾値帯)の
  ローカル変異コマンドを、スナップショット上で実行 → 実際のファイル変更
  diff を効果カテゴリ(repo 外 / 機密パス / control-plane / 削除規模)で
  判定 → 問題なければコミット、危険なら破棄して L4 へエスカレート
- **バックエンド抽象化**: ポータブル基線は git worktree + コピー。
  最適化バックエンドとして overlayfs(Linux)/ APFS clone(macOS)を
  プラガブルに。Windows はまず基線のみ(横断課題のパリティ原則)
- **適用範囲の絞り込み**: hot path 全件には適用しない。L3 が allow/flag を
  確信できるものは素通し(これが L3 の新しい存在意義)。発動条件・予算は
  `policy.transactional`(閾値・タイムアウト・対象ゲート)で制御
- **Assessment の実測化**: 監査ログの reversibility / blastRadius を
  観測値で上書きし、`explain` が「予測 vs 実測」を並べて表示する。
  v0.5 コーパスに「予測と実測の乖離率」指標を追加
- 限界の明文化: スナップショット外への効果(プロセス・IPC・時計)と
  外部送信(L1 の領分)は対象外

**終了条件:** 破壊的コマンドのサンプル群で「実行前に予測で deny」ではなく
「隔離実行の diff を見て判定」が機能し、誤 deny 率がコーパス上で v0.5 比改善。
スナップショット破棄により実害ゼロで危険コマンドを観測できたことを e2e で実証。

## v0.9 — Capability broker & control-plane isolation(L1 完全版)

**テーマ:** サンドボックスランタイムとの統合で deny-all を強制力にし、
belay を「能力拡張の承認ブローカー」として確立する。same-OS-user 問題に着手。

- **サンドボックス統合**: エージェントをネットワークなし・FS スコープ付きの
  サンドボックス(コンテナ / seatbelt / landlock)で起動する構成を公式
  サポートし、belay は能力拡張要求(エグレス開放、repo 外パスへのアクセス)
  の承認ブローカーになる。サンドボックス自体は実装せず、既存ランタイムに
  乗る(ADR-001 Option B との分業)
- **control-plane の信頼ドメイン分離**: 承認状態・ピン・署名鍵を別 OS
  ユーザーまたは read-only マウントの管理下に置く構成を提供し、
  same-OS-user の adversarial エージェントに対する「検出可能」から
  「改竄不能」への格上げ経路を作る(v0.3 ハッシュピン + v0.6 署名トークンの
  完成形)
- **層別 conformance**: v0.4 のアダプタ適合スイートを拡張し、
  「L1 有効 / L2 有効 / L3+L4 のみ」の各構成で保証マトリクスを検証する
- adversarial 耐性は**この構成を有効化した場合のみ**主張する。
  デフォルト(L3+L4)構成の保証は従来どおり cooperative-but-fallible に限定

**終了条件:** サンドボックス構成下で、エージェントが belay の承認なしに
外部送信・repo 外書き込み・control-plane 改竄のいずれも達成できないことを
e2e で実証。各構成の保証差分が文書化されている。

## v1.0 — Stable Belay(保証を層別に約束する)

従来の v1.0 コミットメント(semver 保証のアダプタ SDK、複数アダプタ、
成熟したポリシー言語、v0.x コマンドセットからの移行パス)に加え、
ADR-001 の帰結として:

- **層別 guarantee table の公開**: 「何を・どの層が・どの前提で」保証するかを
  構成(L1 有無 / L2 有無)ごとに明記し、テスト可能・テスト済みにする。
  guardrail を boundary と呼ばない原則の最終形
- **署名・検証された control plane** を標準構成にする(v0.9 の分離構成を
  デフォルト推奨へ)
- L3 リストの位置づけを文書で確定する: 「安全境界ではなくノイズ削減
  キャッシュ」。リスト更新はマイナーリリースで可能(セキュリティ修正扱い
  にしない)という運用ルールを semver ポリシーに含める

**終了条件:** 破壊的変更が semver に従う。guarantee table が層・構成ごとに
テストで裏づけられている。新しいアダプタが公開ドキュメントだけで書ける。
README の denylist 批判が、自家撞着なしに実装の説明として成立している。
