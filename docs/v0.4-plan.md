# v0.4 対応事項 — Portable adapters & デフォルト硬化

対象: v0.4.0
前提: [`docs/v0.3-remaining.md`](./v0.3-remaining.md) の P0/P1(保証の穴)が
v0.3.x で閉じていること。**ゲートは信頼できるようになってから移植する**
(SPEC-v0.3 §1 の方針)。

テーマは 2 つ:

1. **移植性** — Cursor 専用をやめ、"agent-belay" の名前に見合う構造にする
2. **デフォルト硬化** — v0.3 で opt-in として実装した fail-closed /
   control plane を、dogfood の実測を根拠にデフォルトへ反転する

---

## 1. ランタイムレベルのアダプタ契約

現状 `src/adapters/cursor/runtime-entry.ts` は core 関数
(`classifyShell` / `classifySubagent` / `classifyToolUse`)を直接呼び、
Cursor のフックペイロードの癖が事実上の契約になっている。
v0.2 の `BelayAdapter`(`src/adapters/types.ts`)は **installer レベル**のみ。

- 正規化された `GatedAction`(kind, command/payload, cwd, repoRoot,
  contractVersion)を入力、正規化された `Verdict` を出力とする
  **versioned な契約型**を core に定義する
- ペイロードを正規化できないアダプタは **fail closed**(deny)する
- `runtime-entry.ts` をこの契約越しに書き直し、契約の単体テストを置く

## 2. core からホスト前提を除去

core に残る Cursor 前提(lint/テストで再発防止する):

- `src/core/config.ts` — `audit.logPath` デフォルトが
  `.cursor/belay/audit.ndjson`(L132)、`belayStateDir()` の repo-local
  フォールバックが `.cursor/belay`(L531)。これらはアダプタ側へ移す
- `src/adapters/cursor/runtime-entry.ts` の `findRepoRoot()` が
  `.cursor` ディレクトリをリポジトリ判定に使用 — アダプタ責務として明示
- config ファイルパス(`.cursor/belay.config.json`)の解決もアダプタへ

検収: 「core が `.cursor` / `.claude` 等の文字列を含まない」ことを
テストで強制(roadmap の exit criteria)。

## 3. 第二アダプタ — Claude Code hooks

契約が一般化する証明として Claude Code アダプタを実装する:

- `PreToolUse`(Bash / Task)→ shell / subagent / fileMutation ゲート
- `UserPromptSubmit` → `/belay-approve` 承認ラウンドトリップ
- 監査イベントの書き出し(audit.ndjson 相当)
- インストールレイアウトは `.claude/` 配下で Cursor 版
  (`.cursor/hooks/belay-*.mjs` + runtime バンドル)と対称にする
- **control-plane 保護(R8 / ハッシュピン R7)を `.claude/` 配下の
  成果物にも拡張**する — v0.3 で repo 内保護を path 集合化しておくと流用できる

## 4. アダプタ適合(conformance)スイート

両アダプタが**同一のシナリオ集合**を通ることを CI で保証する:

- verdict マトリクス(R16–R18 相当の allow / allow_flagged / deny)
- 承認ラウンドトリップ(one-shot、TTL、revoke)
- fail-closed 配管(hook 例外 → deny、正規化不能 → deny、改竄 → deny)
- redaction(秘密が監査ログに平文で残らない)
- 既存フック・設定との共存(hooks.json / settings へのマージが破壊的でない)

検収: 第三のアダプタが「ハーネスを書いてスイートを通すだけ」で追加できる。

## 5. デフォルト反転(dogfood 実測が条件)

`agent-belay metrics`(OQ1)の would-block 率・top reasons が許容範囲なら:

- `policy.unknownLocalEffect`(および v0.3.x で追加する
  `policy.unparseableShell`)のデフォルトを `"deny"` に反転
- `controlPlane.enabled` のデフォルトを `true` に(repo キー導入済みが前提)
- ハッシュピン(integrity)をデフォルト有効に
- 移行ポリシー: SPEC OQ1 のフォールバック案どおり、
  **新規インストールは deny、v2/v3 からの移行インストールは現行値を維持**
  という分岐を `init`/`upgrade` 時に決める
- README / SECURITY.md / skill テンプレートの記述更新
  (deny が増える分、`explain` と `overrides.allow` への導線を強化)

ノイズが許容できない場合はデフォルト反転だけ v0.5 に送り、
アダプタ作業と切り離してリリースできるようにしておく。

## 6. Windows パリティ

control plane と第二アダプタの両方に効く横断課題:

- `defaultControlPlaneDir()`(`src/core/config.ts` L506)は
  `XDG_CONFIG_HOME` / `~/.config` のみで **`%APPDATA%` 未対応**。
  `homedir()` も `process.env.HOME` 依存(Windows では空)
- `.cmd` ランナーと hook パス解決のパリティを conformance スイートに含める

## 7. 候補(v0.4 に入れるか判断するもの)

- **エージェント側 `Assessment` の取り込み** — `ROADMAP.md` は v0.4+、
  `ROADMAP-strategic.md` は v0.5 に置いている。独立判断を常に併走させる設計が
  必要で classifier 改修と密結合のため、**v0.5(Smarter judgment)に残す**
  ことを推奨。v0.4 では契約型に `assessment?` フィールドの席だけ予約する。
- **ポリシーバンドル / チームプリセット**(`docs/ROADMAP.md` の v0.4+ 項)—
  v0.6 の layered config と重なるため、v0.4 では `overrides` を別ファイルに
  切り出せる程度の最小対応に留めるか見送る。

## 終了条件(roadmap 準拠)

1. core にホスト固有の前提がない(lint/テストで強制)
2. Cursor / Claude Code の 2 アダプタが同一 conformance スイートを通過
3. アダプタ追加に core の分類ロジック変更が不要
4. (実測が許せば)デフォルトインストールが fail-closed +
   out-of-repo control plane で動作する
