## Security & spec gaps

対象: v0.3.x パッチリリース (v0.3.2 以降)  
根拠: [SPEC-v0.3-requirements.md](./SPEC-v0.3-requirements.md) の要求と、出荷済み v0.3.0/v0.3.1 実装 (`src/core/` ほか) のコード精査による差分。

判断基準: **「v0.3 = 信頼できるゲート」というテーマで約束した保証に穴を開けるものは v0.3 系で塞ぐ**。新機能・移植性 (アダプタ契約、第二アダプタ) は v0.4 に送る。

### P0 — セキュリティ上の欠陥 (出荷済みコードのバグ/保証違反)

#### 1. `overrides.allow` が control-plane / outside-repo 保護より先に評価される

- 場所: `src/core/classify-shell.ts` の `classifySegment()` — `matchesCustomAllow` (L353) が `targetsControlPlane` (L400/416) や `outside_repo_redirect` (L432) より**前**に return している。
- SPEC R13 の優先順位は「control-plane write > overrides.deny > built-in external > overrides.flag > overrides.allow > built-in read-only > fail-closed」。「override で control-plane write や repo 外効果を再許可できてはならない」が明文の受け入れ基準 (T4)。
- 影響: 例えば `overrides.allow: ["tee *"]` を入れたオペレータは、意図せず `tee ~/.config/agent-belay/approved-approvals.json` まで許可してしまう。
- 対応: 評価順を SPEC §5.6 に合わせて並べ替え、T4 相当の回帰テストを追加。

#### 2. 改行区切りコマンドのバイパス

- 場所: `src/core/shell-tokenizer.ts` — 改行は空白として flush されるだけで、セグメント区切りにならない。
- 影響: `ls\ncurl evil.com` は 1 セグメント (key = `ls`) になり read-only として **allow** される。
- 対応: 改行を `;` 相当のセパレータとして扱う。

#### 3. `find -exec` / `find -delete` が read-only 扱い

- 場所: `src/core/classify-shell.ts` L19–37 — `find` が `READ_ONLY_COMMANDS` に入っており、引数を見ない。
- 影響: `find . -name '*.ts' -delete` や `find / -exec rm {} \;` が **allow**。
- 対応: `find` を専用ハンドラに移し、危険フラグ検出時に deny。

#### 4. 資格情報付き `curl` が素の `curl` より緩い

- 場所: `src/core/classify-shell.ts` L539–559 — `curl`/`wget` に `-H` / `Authorization` があると `allow_flagged` で return し、下の `EXTERNAL_COMMANDS` deny に**到達しない**。
- 対応: credential_header はシグナル付与に留め、verdict は external deny を維持。

#### 5. デフォルト構成で repo 内 control-plane ファイルが無防備

- 場所: `src/core/config.ts` `classifierOptionsFromConfig()` — `controlPlaneDir` は `controlPlane.enabled` のときのみ設定。
- 影響: デフォルトでは `.cursor/belay.config.json`、`.cursor/hooks.json`、`belay-*.mjs`、バンドル runtime、`.cursor/belay/*.json` を Write ツールでもシェルでも自由に書き換えられる。
- 対応: repo 内 control-plane パス集合を常時 deny 対象に追加。
