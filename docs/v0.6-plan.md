# v0.6 対応事項 — Observable & manageable(チームで運用できるゲートへ)

対象: v0.6.0
前提: [`docs/v0.5-plan.md`](./v0.5-plan.md) の判定パイプライン
(ポリシー as code、較正済み confidence、コーパス測定)が出荷済みであること。

## なぜ v0.6 か

v0.5 までで「個人の 1 リポジトリで信頼できるゲート」は完成する。
しかし fail-closed デフォルト(v0.4)とポリシー化(v0.5)の帰結として
**deny とポリシー調整の量が増える**。監査ログが生 NDJSON のまま、承認手段が
エディタ内の `/belay-approve` だけでは、運用負荷がゲート無効化の動機になる —
ROADMAP の横断原則「ノイズの多いゲートは切られる」への回答が v0.6。

成果物はすべて v0.3.1 の `metrics` CLI と audit.ndjson スキーマ
(`mode` / `wouldBlock` / `permission` / `Assessment`)の延長線に置く。

---

## 1. 監査ツーリング — `agent-belay audit`

`audit.ndjson` を「読むファイル」から「問い合わせる記録」にする。

- **query / filter**: 期間・verdict・reason・gate 種別・fingerprint で絞り込み
  (`--json` 対応、`metrics` CLI の集計コードを共有)
- **summarize**: 「このエージェントは何をし、私は何を承認したか」を
  人間向けに要約(deny → 承認 → 実行のラウンドトリップを 1 行に畳む)
- **replay**: 記録済みエントリを現在の設定・分類器に再投入し、
  当時と今で verdict が変わるかを表示(v0.5 のコーパスハーネスを流用)
- **バイパス試行の表面化**: 同一 fingerprint の deny 直後に類似コマンドの
  言い換え(ラッパー化・パス化・分割)が試行されたパターン、および v0.5 の
  自己評価不一致シグナルを抽出してハイライトする
- **ノイズの多いルールの特定**: deny 後すぐ承認されている率が高い
  reason / ルールを報告し、`overrides` / ポリシー調整の候補として提示

## 2. ポリシーシミュレーション

設定・ポリシー変更を本番に当てる前に履歴に対して dry-run する。

- `agent-belay simulate --config <candidate>`: 記録済み audit 履歴に候補設定を
  適用し、「verdict が変わるエントリ」の差分(allow→deny / deny→allow)を出力
- fail-closed 系デフォルト変更(v0.4 の反転で経験済み)のような
  破壊的調整を、各リポジトリが自分の履歴で事前評価できるようにする
- exit criteria 直結: 「設定変更はロールアウト前にシミュレートできる」

## 3. 共有・階層化設定

チームのデフォルト + リポジトリごとの上書き。

- 優先順位を明文化: **組み込みデフォルト < チーム設定
  (control-plane 配下 `~/.config/agent-belay/team.config.json` または
  リポジトリ指定の共有ファイル)< リポジトリ設定 < (上書き不可の)保護層**
- 保護層 = control-plane / outside-repo 保護と integrity 設定。
  チーム設定でも貫通できないことを v0.3 R13 / v0.5 ルール評価器の
  不変条件として継承する
- `doctor` が有効設定の出所(どの層から来た値か)を表示する
- v0.4 で見送った「ポリシーバンドル / チームプリセット」はここで吸収:
  名前付きプリセット(例: `strict` / `standard` / `audit-first`)を
  チーム設定として配布できる形にする

## 4. 帯域外(out-of-band)承認

長時間・無人運転のエージェントで、エディタに戻らず承認できるようにする。

- **署名付き承認トークン**: 承認は control plane の鍵で署名された
  トークンとして発行・検証し、ファイル書き込みだけでは偽造できない形にする
  (v0.3 で承認状態を repo 外へ出し、ハッシュピンを入れた延長。
  guardrail → boundary 路線の v1.0「signed/verified control plane」への布石)
- **通知チャネル**: deny 発生を webhook / コマンドフックで外部へ通知し、
  CLI(リモートシェル経由)またはチャット連携から
  `agent-belay approve <id>` で承認できるようにする
- one-shot / TTL / revoke / 監査記録のセマンティクスは R21 のまま変えない
- スコープ注意: チャット連携そのものはプラガブルな通知 + 承認 API までを
  v0.6 とし、特定 SaaS 向け実装は同梱しない

## 5. メトリクスの拡充

v0.3.1 `metrics`(OQ1 用)を運用ダッシュボードの土台に育てる。

- allow / flag / deny 率の推移(期間バケット)
- **承認レイテンシ**(deny 発行 → 承認/失効までの時間)— 運用負荷の主指標
- top escalation(reason 別・fingerprint 別)とノイズルール候補(§1 と共有)
- バイパス試行カウント(§1 の検出結果)
- `--json` を安定スキーマ化し、外部ダッシュボードへ流せるようにする

---

## 終了条件(ROADMAP 準拠)

1. オペレータが生 NDJSON を読まずに「エージェントが何をして、何を承認したか」
   に答えられる(`audit` の summarize / query で実演テスト)
2. 設定・ポリシー変更がロールアウト前に履歴に対してシミュレートできる
3. チーム設定とリポジトリ設定の優先順位が文書化・テスト済みで、
   保護層はどの層からも貫通できない
4. エディタ外から承認した場合も one-shot / TTL / 監査のセマンティクスが
   保たれ、署名のないトークンは拒否される
5. 承認レイテンシと deny 率が計測でき、v0.4 のデフォルト反転後の
   運用負荷が数値で語れる
