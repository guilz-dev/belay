# v0.5 対応事項 — Smarter judgment(判断を本物にする)

対象: v0.5.0
前提: [`docs/v0.3-remaining.md`](./v0.3-remaining.md) の保証の穴が塞がり、
[`docs/v0.4-plan.md`](./v0.4-plan.md) のアダプタ契約 + 第二アダプタ +
fail-closed デフォルトが出荷済みであること。

## なぜ v0.5 がプロダクトの正念場か

README の核心の主張は「コマンド名ではなく、可逆性・外部効果・影響範囲・確信度の
**実行時判断**で決める」。しかし v0.4 時点でも実態は静的コマンドセット 3 つ
(`READ_ONLY_COMMANDS` / `FLAGGED_COMMANDS` / `EXTERNAL_COMMANDS`)による辞書引きで、
`Assessment` の 4 次元は**ルールごとにハードコードされた定数ラベル**にすぎない。
confidence(0.61〜0.99 の手書き定数)が verdict を駆動する仕組みは存在しない。

v0.5 はこのギャップを閉じる。**これが終わるまで「静的 denylist を超えた」とは
言えない**。逆に言えば、v0.5 が完了して初めて README の第一段落が事実になる。

## 1. 判定コーパスと測定基盤(最初にやる)

現状、「判断が良いか」を主張する根拠が**ゼロ**。改善の前に測定を置く。

- **コマンドコーパス**: 実コマンド + 期待 verdict(+ 期待 reason)のラベル付き
  データセットを `corpus/` に整備する。ソースは
  (a) dogfood の `audit.ndjson` から匿名化抽出(`metrics` CLI を拡張)、
  (b) セキュリティレビューのバイパス事例、
  (c) READ_ONLY / FLAGGED / EXTERNAL の既存セットの境界事例
- **`pnpm corpus` 評価ハーネス**: precision / recall / FPR(誤 deny 率)を
  verdict 別に出力し、CI でベースラインからの劣化を fail にする
- サブエージェント分類(R11 で作ったテキストコーパス)も同じハーネスに統合
- 以降の v0.5 の全変更は「コーパス指標が改善し、v0.3/v0.4 スイートが
  退行しない」ことを受け入れ条件とする(ROADMAP の exit criteria)

## 2. 4 次元を「飾り」から「判定入力」へ

判定パイプラインを「コマンド名 → verdict 直結」から
「構造化属性 → 4 次元評価 → 閾値 → verdict」へ再構成する。

- **引数レベル解析**: 同じコマンド名でもフラグで次元が変わることを表現する
  - `rm` vs `rm -rf`、`git push` vs `git push --force`、
    `git checkout -- .`(作業ツリー破壊)、`chmod -R`、`docker run --privileged`
  - 対象パス数・glob の広がりから blast radius を「ラベル文字列」ではなく
    **構造化値**(対象スコープ: file / dir / repo / outside / external)にする
- **reversibility の実計算**(最小セット):
  - 対象が git 管理下か(`git ls-files` 結果のキャッシュ)→ 上書き・削除は
    `recoverable_with_cost` に緩和、未追跡ファイル削除は格上げ
  - append(`>>`)と truncate(`>`)の区別
- **calibrated confidence**: コーパスで較正した確信度を出力し、
  `policy.confidenceThresholds`(allow / flag / escalate の境界)が
  verdict を駆動する。手書き定数は較正値に置換する
- `Assessment.signals` は v0.2 からの種(seed)— ここに引数レベルの
  シグナルを足し、`explain` で人間が追える形を維持する

## 3. ポリシー as code

場当たり的なコマンドセット(`overrides` 含む)を、構造化属性に対する
宣言的ルールへ置き換える。

- ルール形式: `match`(コマンド key / 引数属性 / パススコープ / 4 次元値)
  → `action`(allow / flag / deny / escalate)
- 組み込みデフォルトポリシー自体を同じルール言語で記述し、
  **単体でテスト・レビュー可能**にする(ROADMAP exit criteria)
- v0.3 の `overrides.{allow,flag,deny}` は最も単純なルールとして
  自動マイグレーションし、後方互換を維持する
- R13 の優先順位保証(control-plane / outside-repo はルールで貫通不可)を
  ルール評価器のレベルで構造的に強制する

## 4. オプションのモデル支援分類(曖昧な中間層のみ)

- ヒューリスティクスが低確信(較正済み confidence が閾値帯)のときだけ、
  設定で有効化された場合に限り LLM へ 4 次元の判定を諮る
- **デフォルトは無効**。モデル未設定・呼び出し失敗時はヒューリスティクス
  verdict に決定論的にフォールバックする(fail-closed 方針と整合:
  escalate 側に倒す)
- 実装時は最新の Claude モデルをデフォルトにする
- プロンプト・応答は監査ログに redaction 適用済みで記録し、
  モデル判定が効いたことを `signals` に残す
- レイテンシ予算: 同期フックの hot path にあるため、タイムアウト時は
  ヒューリスティクス verdict(コーパスで計測した予算は cross-cutting backlog
  のパフォーマンス項を参照)

## 5. エージェント側 `Assessment` の取り込み(abstract Belay substrate)

v0.4 で契約型に予約した `assessment?` フィールドを実装する。

- ホストがエージェント自己評価を渡してきた場合、**独立判断を必ず併走**させ、
  自己評価は「確信度の補強」または「不一致シグナル」としてのみ使う
- 自己評価と独立判断が食い違う(エージェントは安全と言うが Belay は危険と
  判定)場合は escalate し、不一致を監査ログに記録する —
  これは v0.6 のバイパス試行サーフェスの入力になる
- 監視対象の自己申告を verdict の根拠にしない、という不変条件をテストで固定

## 終了条件(ROADMAP 準拠 + 測定の追加)

1. コーパス上の precision / recall が v0.3/v0.4 ベースラインを上回り、
   既存スイートに退行がない(CI で強制)
2. モデル支援は完全にオプションで、無効・失敗時は決定論的に劣化する
3. ポリシールールが単体でテスト可能で、組み込みデフォルトも同じ言語で
   記述されている
4. confidence が手書き定数でなく較正値であり、閾値設定が verdict を変える
   ことがテストで示せる
5. README の「reversibility / external effects / blast radius / confidence で
   判断する」という記述が、実装の動作として正確になっている
