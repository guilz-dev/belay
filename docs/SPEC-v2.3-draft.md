# agent-belay SPEC v2.3（草案）— skill としての belay の立ち位置（命綱の完成）

Status: **Draft / exploration**（v2.2 実装後に再評価する位置づけメモ。実装着手の合意ではない）
Builds on: SPEC-v2.2（skill=front door / hook=floor の二層）/ CONCEPT-v2.0（restorability floor）
扱う問い: v2.2 で薄い「配布の表口」だった skill を、**配布以上の価値を持つ助言知性層**に
できるか。belay の思想（復元可能性の床 / FN=0 / fence ではない）を壊さずに。

---

## 0. 立ち位置の転換（v2.2 → v2.3）

- v2.2 の skill は **必要だが知性の薄い front door**（distribution + bootstrap）。判定も強制も
  hook（floor）が担い、skill は「導入と承認の案内」に徹していた。
- v2.3 の機会は「**skill が hook の上に乗る助言知性層になる**」こと。**floor は止める。skill は
  考えさせ・引き上げ・見せる。**
- 名前の由来（climbing **belay** = 落下を受け止める命綱）が指す本来像は、「止める」だけでなく
  **「落ちた時に引き上げる」**まで含む。v2.3 はこの比喩を完成させる。

> 不変条件（最重要・非交渉）: **skill は floor を代替しない。** skill が「ルールを助言する fence」
> に化けると belay の核（denylist/fence は破綻）に逆行する。v2.3 のすべての機能は
> **floor=hook を前提に、その上に積む助言/UX/復元**であって、強制の置き換えではない。

## 1. 三本柱

### 柱① 下流の復元支援（**最有力・命綱の本来像**）

何かが起きた／起きかけた後、skill が**復元を助ける**。

- 監査ログ（observed audit）と verdict から、直近の破壊的操作を特定し、**具体的な戻し方**を提示:
  `.env` を消した → `git restore` / `git reflog` / スナップショット、DB を落とした → バックアップ
  からの復旧手順、外部 publish した → ロールバック可否の説明。
- belay が `ask` で止めた操作については、「なぜ止めたか + 安全な代替（バックアップ→実行）」を
  併せて出す。
- **思想的整合**: restorability floor は「取り消せるか」で判定する。その判定主体が、いざ取り消す
  段で**取り消し方を知っている**のは自然。「止める床」と「引き上げる命綱」の二役が belay の独自性。
- **境界**: 復元の実行（`git restore` 等）はエージェント/人間が行う。skill は手順を**助言**する
  だけで、勝手に復元コマンドを撃たない（破壊的逆操作も hook の対象）。

### 柱② 上流の復元可能性ディシプリン（エージェントの計画を形成）

skill が読み込まれると、エージェント自身が**行動前に「取り消せるか?」を自己点検**する。

- 破壊的操作の前に、可逆な代替を**先に提案**する癖を与える（バックアップ→実行、`--dry-run`
  先行、`rm` の前に `mv`退避 等）。
- belay の restorability レンズをエージェントの**思考プロセス**に持ち込む（hook の手前で）。
- **境界（要注意）**: これは助言なのでモデルが無視しうる。**floor と誤認させない**設計が必須
  （あくまで hook が硬い床。skill は上流の癖付けで、止める保証はしない）。

### 柱③ 床の可視化（信頼を作る）

belay の弱点は「98% を黙って通す」= **効いているのが見えない**こと。

- skill が監査を定期要約: 「今週 belay は破滅的操作を N 件ブロック、M 件フラグ、98% は素通り」。
- 不可視の床を**信頼できる相棒**に変える（`/belay why`（R-S3）の延長）。
- **導入障壁が低く早く出せる** → 信頼構築の入口として v2.3 の先頭に置ける。

### 補助柱

- **④ 自然言語ポリシー作成**: 「prod には一切触らせたくない」→ `overrides` / `sensitivePaths`
  を生成・提案（適用は明示）。
- **⑤ 承認履歴の cross-agent 携行**: skill が承認/設定を携行し、Cursor/Claude/Codex で一貫した
  挙動にする（skill のクロスエージェント性を活かす）。

## 2. アーキテクチャ原則（v2.3）

三層を明文化する:

| 層 | 役割 | v2.3 での担当 |
| --- | --- | --- |
| **hook = floor** | 止める（強制・非バイパス） | 変更なし（v2.0〜v2.2 のまま） |
| **skill = 助言知性** | 考えさせる（上流）/ 引き上げる（下流）/ 見せる（可視化） | 柱①②③ |
| **CLI = 機構** | 判定・配置・監査の実体 | skill は CLI を呼ぶだけ（v2.2 R-S3 の延長） |

- skill の新機能も **判定ロジックを二重実装しない**（R-S1 を継承）。復元手順・要約・ディシプリン
  はいずれも CLI/audit を入力にした**助言の生成**であり、新しい停止/緩和を導入しない。

---

## 3. 非ゴール

- floor の代替・ephemeral トグル（v2.2 §4 を継承）。
- skill による自動復元コマンドの実行（破壊的逆操作も hook 対象。skill は助言のみ）。
- 判定ロジック・三値 boolean・Tier0/Tier1 の変更。

## 5. v2.2 との関係

- v2.2（skill front door + 横展開 + scope）の **実装完了が前提**。skill が確実に floor を
  ブートストラップし、scope/trust が安定した上で、その skill に知性を積むのが v2.3。
- v2.2 の `/belay why`（R-S3）と doctor 可視化が、柱③（可視化）の素地になる。

## 6. 推奨の入口

- **柱①（復元支援）を v2.3 の軸**に据える。名前の比喩・restorability floor の思想・
  「fence ではない」の全部に乗り、belay の独自性（止める床＋引き上げる命綱）を確立する。
- ただし**柱③（可視化）を先に小さく出す**と、低障壁で信頼を作りつつ ①②へ繋げられる。
